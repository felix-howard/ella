# Brainstorm Report: Document-First Client Workflow Redesign

**Date:** 2026-01-27
**Branch:** feature/multi-tax-year
**Status:** APPROVED - Ready for Implementation Planning

---

## Problem Statement

Ella's current client management workflow is **questionnaire-driven**, requiring staff to complete detailed intake forms upfront when creating clients. This conflicts with real-world usage patterns:

1. **Staff friction:** Don't want to answer detailed questionnaire before documents arrive
2. **Client behavior:** Clients rarely answer tax organizers - they just send documents
3. **Workflow mismatch:** System expects specific documents based on questionnaire, but AI can classify independently
4. **Multi-year underutilized:** TaxEngagement model exists but workflow doesn't leverage year-over-year data

### Staff Feedback (Direct Quotes)
> "Staff want the client creation to be simple, just name and phone, then send a message to client"
> "Want a file explorer tab, with all the docs that client send, have AI to rename file, classify file"
> "The tax organizer/questionnaire will be handled later because clients don't actually answer tax organizer"

---

## Evaluated Approaches

### Option A: Modify Existing Pages
**Pros:** Less code, faster initial delivery
**Cons:** High risk of conflicts, questionnaire logic deeply embedded
**Verdict:** REJECTED - Too risky, staff feedback indicates fundamental approach is wrong

### Option B: Fresh Pages with Route Prefix (SELECTED)
**Pros:** Clean slate, no conflicts, can test independently, easy rollback
**Cons:** Temporary code duplication
**Verdict:** SELECTED - Best for quality and maintainability

### Option C: Feature Flag Toggle
**Pros:** Single route, toggle between old/new
**Cons:** Complex conditional rendering, harder to debug
**Verdict:** REJECTED - Adds unnecessary complexity during development

## Implementation Considerations

### Risks
1. **AI classification without questionnaire context:** May have slightly lower accuracy
   - Mitigation: AI already works independently, questionnaire just adds hints
2. **Data migration:** Existing clients have questionnaire data
   - Mitigation: Data preserved, V2 reads same TaxEngagement model
3. **Staff training:** New workflow different from current
   - Mitigation: Simpler flow, intuitive file-based approach

### Dependencies
- Existing TaxEngagement model (no changes needed)
- Gemini classification service (already works independently)
- Twilio SMS service (already integrated)

### Not In Scope (Future)
- Mobile app version
- Bulk client import
- Client self-service questionnaire (portal-side)
