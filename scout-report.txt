# Scout Report: SMS/Welcome Message Flow
Date: 2026-03-30

## BACKEND SMS SENDING FLOW

Core Files:
1. apps/api/src/services/sms/message-sender.ts (line 58-104: sendWelcomeMessage function)
2. apps/api/src/services/sms/twilio-client.ts (sendSms with retry logic)
3. apps/api/src/routes/clients/index.ts (line 446: triggers welcome SMS on client creation)
4. apps/api/src/routes/messages/index.ts (message API endpoints)
5. apps/api/src/routes/webhooks/twilio.ts (line 166-225: SMS status webhook)

Key Flow:
- sendWelcomeMessage() formats message
- sendAndRecordMessage() sends SMS FIRST via Twilio
- ONLY creates DB message record if SMS succeeds (prevents false "sent" status)
- Updates conversation.lastMessageAt and taxCase.lastContactAt
- Twilio returns SID and status ("queued")
- Returns { success, messageId, smsSent, error }

Database Model (Message table):
- twilioSid (unique) - Twilio message ID
- twilioStatus - Delivery status
- channel - SMS/PORTAL/SYSTEM/CALL
- direction - OUTBOUND/INBOUND
- templateUsed - "welcome", "missing_docs", etc
- sentById - Staff sender

Status Values:
- Success: queued, sent, delivered
- Failure: failed, undelivered
- Error format: "failed:21612:Invalid phone number"

Twilio Retries:
- 2 retries on transient failures
- Exponential backoff: 500ms, 1000ms
- Non-retryable errors: invalid phone, landline, permission denied, unsubscribed

## TWILIO STATUS WEBHOOK

Endpoint: POST /webhooks/twilio/status (apps/api/src/routes/webhooks/twilio.ts line 166-225)

Receives from Twilio:
- MessageSid
- MessageStatus (queued, sent, delivered, failed, undelivered)
- ErrorCode (if failure)
- ErrorMessage (if failure)

Processing:
1. Validate Twilio signature
2. Find Message by twilioSid
3. Update twilioStatus field
4. Format: "{status}:{errorCode}:{errorMessage}" if error
5. Return 200 OK to prevent Twilio retry

## FAILURE SCENARIOS

Scenario A: SMS Send Fails Before Recording
- sendSms() fails (invalid phone, no permission, etc)
- Message record NOT created
- Response: { success: true, smsSent: false, error: 'TWILIO_ERROR_21612:...' }
- Conversation exists but shows "No messages"
- User sees conversation in inbox but can't open it with content

Scenario B: SMS Sent But Delivery Fails
- Message sent to Twilio, returns SID
- Message recorded with status='queued'
- Twilio attempts delivery, fails
- Webhook updates: twilioStatus='failed:21612:Invalid phone'
- Frontend displays: X icon + "Sending failed" + expandable error details
- Message remains visible in conversation

Scenario C: SMS Success
- Message sent, SMS recorded with status='queued'
- Twilio webhook updates: queued → sent → delivered
- Frontend shows status progression
- Message visible with "Delivered" status

## IMPLEMENTATION DETAILS

Race Condition Prevention:
- Conversation created with lastMessageAt=new Date() at client creation
- Ensures conversation visible in inbox immediately
- When SMS succeeds, lastMessageAt updated to actual message timestamp
- Code comment: "SMS will overwrite lastMessageAt with correct timestamp"

Unique Constraint:
- twilioSid is unique on Message table
- Prevents duplicate records from webhook retries

Phone Formatting (formatPhoneToE164):
- Strips all non-digits except leading +
- Assumes US (+1) if no country code
- Removes leading 1 from 11-digit US numbers
- Examples:
  * "5551234567" → "+15551234567"
  * "+1-555-123-4567" → "+15551234567"
  * "+84912345678" → "+84912345678" (Vietnam preserved)

Twilio Configuration:
- isTwilioConfigured() checks TWILIO_* env vars
- If not configured: SMS disabled, no error
- Conversation still created
- Response: smsStatus: { sent: false }
